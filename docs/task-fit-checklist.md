# このフローに向く task / 向かない task のチェックリスト

mysk は bounded task 向けです。向き不向きを先に見た方が成功率が上がります。

## 向く task

- 変更対象ファイルが少ない
- scope を spec で明示しやすい
- acceptance を観測可能な文で書ける
- 最初に見るファイルとテストを絞れる
- docs/text-only か、小さめの code change に収まる

## 向かない task

- repo 全体の設計変更が必要
- DB schema や migration を広く触る
- 依存先が多く、最小確認対象を絞れない
- acceptance が曖昧で、人ごとの解釈差が大きい
- 手作業オペレーションや外部環境依存が強い

## 迷ったときの質問

1. 最初に開く 3 ファイル以内へ絞れるか
2. 変更してよいパスを短く書けるか
3. 実装完了を `受け入れ条件` で判定できるか
4. repo 全体探索なしでも着手できるか

3 つ以上 `はい` なら、このフローに乗せやすいです。`いいえ` が多いなら、先に設計整理や人手判断を入れた方が安全です。
