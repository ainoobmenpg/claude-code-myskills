# Issue Draft: `/mysk-implement-start` を粗い `impl-plan.md` でも回るように補強する

## 背景

現状のワークフローでは、`/mysk-spec-implement` が `spec.md` から `impl-plan.md` を生成し、`/mysk-implement-start` がその計画を読んで実装を進める。

ただし実際には、以下の理由で `impl-plan.md` の指示だけでは実装品質が安定しない。

- 仕様書だけでは、正確なファイルパス・関数名・行番号まで確定できないことが多い
- 弱いモデルほど、曖昧なタスクから適切な探索や再分解を自力で行いにくい
- 現在の `mysk-implement-start` は「impl-plan を読む」以降の補完手順が薄く、微妙な計画をそのまま実装しにいきやすい

結果として、「計画の粒度を厳しくする」だけでは十分でなく、「粗い計画でも実装段階で補正できる」指示が必要になっている。

## 問題

### 1. `impl-plan.md` の詳細度を upstream に寄せすぎている

- `mysk-spec-implement` は `spec.md` 読み込み主体のコマンドである
- それにもかかわらず、ファイルパス・変更箇所・擬似コードまで必須化すると、根拠の弱い“もっともらしい実装指示”を固定化しやすい

### 2. `mysk-implement-start` に補完アルゴリズムが不足している

- 曖昧なタスクをどう再分解するかが弱い
- 既存コードから類似実装を探す手順が弱い
- `impl-plan.md` と `spec.md` と実コードが食い違ったときの判断順序が弱い
- 着手前に「最小変更でどこから触るか」を決める流れが弱い

### 3. 周辺ドキュメントとテストが新方針に追従していない

- `mysk-workflow.md`
- `README.md`
- `tests/integration/spec-workflow-mock.bats`

このあたりが旧来の粗い plan 例のままだと、利用者にもモデルにも矛盾した期待を与える。

## ゴール

- `impl-plan.md` が荒くても、`/mysk-implement-start` 側で探索・補完・再分解して前に進めるようにする
- `mysk-spec-implement` は「不確実な精密さ」を強要せず、実装時に役立つ手がかりを安定して残す方向に調整する
- ドキュメント、ワークフロー、テストの期待値を揃える

## 非ゴール

- `mysk-spec-implement` にフルコードリーディング責務を持たせること
- すべてのケースで厳密なファイル/行番号を事前確定すること
- 実装計画だけで repo 探索を完全に不要にすること

## 設計方針

### A. `impl-plan.md` は「ヒント」であって「唯一の真実」ではない

`/mysk-implement-start` は以下の優先順位で判断する。

1. ユーザーの明示指示
2. 現在のコードベースの実態
3. `spec.md` の要求
4. `impl-plan.md` の詳細記述

`impl-plan.md` は着手順序や意図のガイドとして使うが、コードベース実態と矛盾する詳細は盲信しない。

### B. 実装前にプリフライトを必須化する

各フェーズ開始前、または各タスク着手前に最低限以下を行う。

1. `impl-plan.md` の対象タスクを読む
2. `spec.md` の対応箇所を読む
3. repo 内で関連ファイル・類似実装・既存パターンを探す
4. 「今回触る最小単位のファイル/関数/テスト」を決める
5. その結果を踏まえて、必要ならタスクを実行可能な粒度に再分解する

### C. 粗いタスクは実装開始前に自己再分解する

`impl-plan.md` が以下の状態なら、そのまま着手せず再分解する。

- 対象ファイルが曖昧
- 変更箇所が広すぎる
- 受け入れ条件との対応が弱い
- 擬似コードが抽象的すぎる

再分解時は最低でも以下を埋める。

- 触る候補ファイル
- 確認すべき既存実装
- 追加/修正する関数や責務
- 先に書くべきテスト or 検証方法
- このタスクを完了と見なす条件

### D. 弱いモデル向けに探索手順を明文化する

`/mysk-implement-start` に以下のような補助手順を入れる。

- ファイル探索: `rg --files`, `rg -n`
- 類似実装探索: 同名/類似機能/既存パターンを探す
- 呼び出し元探索: 変更対象がどこから使われるかを見る
- テスト探索: 関連テストの有無を確認する
- 既存規約確認: 近いディレクトリのコードスタイルや責務の切り方を確認する

### E. `mysk-spec-implement` は「確度つき計画」に寄せる

`mysk-spec-implement` では、コードを読まずに断定しにくい情報を必須扱いしない。

代わりに、各タスクに以下のような情報を持たせる。

- `対象ファイル`: 確定 or 候補
- `変更箇所`: 確定 or 調査必要
- `実装メモ`: 実装時に確認すべき点
- `探索キーワード`: `mysk-implement-start` が repo 探索時に使える語
- `対応する受け入れ条件`

## 実装タスク

### 1. `commands/mysk-implement-start.md` を補強する

- 「実行ルール」にプリフライト手順を追加する
- 「impl-plan.md の解釈ガイドライン」に、曖昧タスクの再分解ルールを追加する
- `impl-plan.md` と `spec.md` と repo が食い違った場合の判断順序を追加する
- 類似実装探索、関連テスト探索、最小変更方針の選び方を追加する
- タスク未解像時の fallback を追加する
  - `spec.md` を再読
  - repo 探索
  - 実行可能タスクへの再分解
  - それでも不明なら質問
- フェーズ実行時に「まず計画をそのまま実行する」のではなく、「必要に応じて task を実装可能な粒度へ補正する」と明文化する

### 2. `commands/mysk-spec-implement.md` を調整する

- 「コードブロックレベルまで分解」を維持する場合でも、未確認情報を断定必須にしない形へ変更する
- 行番号・関数名・ファイル名は「確認できる場合は書く」「未確認なら候補/調査メモで残す」に変える
- `mysk-implement-start` が利用しやすい補助情報を追加する
  - 探索キーワード
  - 類似実装の当たり先
  - 実装前に確認すべきこと
- Markdown の壊れたコードフェンスを修正する

### 3. `commands/mysk-workflow.md` を同期する

- ステップ4の実装計画説明を新方針に合わせる
- ステップ5相当で「計画が粗くても実装開始時に探索・補完する」ことを明記する
- ワークフロー全体として、plan の粒度と implement 側の補完責務の分担を説明する

### 4. `README.md` を同期する

- `/mysk-spec-implement` の出力説明を見直す
- `/mysk-implement-start` の説明に「impl-plan を補完しながら実装する」ことを反映する
- 必要なら簡単な flow 例を追記する

### 5. テストと fixture を更新する

- `tests/integration/spec-workflow-mock.bats` の `impl-plan.md` fixture を新方針に合わせる
- 旧フォーマットだけを前提にした説明があれば修正する
- 必要なら以下の観点を追加する
  - `impl-plan.md` が粗くても許容される
  - `impl-plan.md` に候補ファイルや調査メモが含まれても許容される
  - `mysk-implement-start` の期待フォーマットが「断定情報のみ必須」になっていない

## 受け入れ条件

- `mysk-implement-start` の仕様に、repo 探索と自己再分解の手順が明記されている
- `mysk-implement-start` の仕様に、判断順序（ユーザー指示 / repo / spec / impl-plan）が明記されている
- `mysk-spec-implement` の仕様が、不確実なファイル名や行番号の断定を必須にしていない
- `mysk-spec-implement` の出力例に、候補情報や調査メモを含められる
- Markdown サンプルが正しくレンダリングされる
- `mysk-workflow.md` と `README.md` の説明が新方針に一致している
- テスト fixture が新しい期待値と矛盾しない

## 補足メモ

今回の修正で狙うべきは「plan をさらに厳密化すること」よりも、「plan が不完全でも implement 側で回収できること」。

そのため、主戦場は `mysk-spec-implement` ではなく `mysk-implement-start` 側に置くべきである。
