# spec.md の読み方

`spec.md` は、このフローで実装者と reviewer が最初に見る一次資料です。迷ったときは repo の憶測ではなく、まず `spec.md` を基準に判断します。

## どこから読むか

1. `## 概要`
2. `## 最小確認対象`
3. `## スコープ`
4. `## 制約条件`
5. `## 受け入れ条件`

これで「何を変えるか」「最初に何を見るか」「どこまで触ってよいか」が分かります。

## セクションの意味

- `概要`: その task が何を達成したいかの短い説明です。
- `目的`: なぜその変更が必要かです。優先順位で迷ったときの判断軸になります。
- `利用者` / `ユースケース`: 誰がどう使う前提かを示します。
- `入出力`: 振る舞いの境界です。API、CLI、文言変更の期待値を確認します。
- `最小確認対象`: 最初に読むファイル、テスト、コマンドです。bounded task ではここから外へ広げないのが基本です。
- `スコープ > 範囲内`: 変更してよいパスや対象です。
- `スコープ > 範囲外`: 触らない前提です。
- `制約条件`: 守るべき既存仕様、非互換禁止、出力形式などです。
- `受け入れ条件`: pass / fail を決める観測可能な条件です。
- `リスクと未決定事項`: 実装前に確認が要る点です。
- `タスク種別`: `docs/text-only` か `code` かの判定です。review / verify の重さに影響します。

## `最小確認対象` の見方

ここには「最初に見る working set」が書かれます。たとえば:

```markdown
## 最小確認対象
- `commands/mysk-review.md`: 初回 review 起動文の確認
- `tests/unit/benchmark-review-context.bats`: 起動文の契約テスト
```

この場合は、まず 2 ファイルだけ読めばよいという意味です。ここで十分な根拠が取れたなら、repo 全体へ探索を広げません。

## `受け入れ条件` の見方

`受け入れ条件` は「実装後に何を確認すれば完了か」を表します。実装中は TODO、review では検査項目、verify では証跡の元になります。

見方のコツ:

- 文言が観測可能かを見る
- 「不要な変更がないこと」のような制約が、必要な変更まで禁止していないか確認する
- docs task では、変更後に残る literal な文言が書かれているかを見る

## 小さな例

```markdown
## 最小確認対象
- `README.md`: 実際の案内文
- `docs/workflow.md`: 詳細フロー

## 受け入れ条件
1. README から新しいガイドに辿れる
2. `docs/workflow.md` からも同じガイドに辿れる
3. 既存の公開コマンド説明は壊さない
```

この spec なら、次の行動は明確です。まず `README.md` と `docs/workflow.md` にリンクを追加し、その後にリンク切れや説明崩れがないか確認します。

## 次にどう動くか

- 実装前: `最小確認対象` を開く
- 実装中: `スコープ` と `制約条件` から外れていないか見る
- 実装後: `受け入れ条件` を 1 件ずつ潰す
- 判断に迷う: `リスクと未決定事項` を見て、必要なら spec を直す
