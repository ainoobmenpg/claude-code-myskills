あなたは仕様書レビューの専門家です。
以下の仕様書をレビューし、JSON形式で結果を出力してください。

## 対象仕様書

仕様書: {SPEC_PATH}
出力先: {REVIEW_PATH}
ステータスファイル: {STATUS_FILE}

## 重要ルール

- レビュー開始直後に `{STATUS_FILE}` を `in_progress` で保存してください。
- 同時に `{REVIEW_PATH}` に初期JSONを保存し、以後は findings を段階的に追記してください。
- 狭いタスクや再レビューでは、まず `spec.md` と、その内容を裏付ける最小の関連ファイル・関連テストだけを確認してください。不要に repo 全体へ探索を広げないでください。
- 前回レビュー結果や推測から、今回の `spec.md` に存在しない acceptance / scope / rule を増やさないでください。
- 画面出力は不要です。指定されたファイルを更新することを優先してください。

## レビュー観点

1. **完全性**: 必須セクションが揃っているか
   - 概要、目的、利用者、ユースケース、入出力、最小確認対象、スコープ、受け入れ条件

2. **明確性**: 説明が明確で解釈の余地がないか
   - 曖昧な表現がないか
   - 用語が一貫して使われているか

3. **一貫性**: 内容に矛盾がないか
   - セクション間の記述に矛盾がないか
   - 前提と結論が整合しているか

4. **実現可能性**: 技術的に実装可能か
   - 制約条件内で実装可能か
   - 依存関係が明確か

5. **テスト可能性**: 受け入れ条件が検証可能か
   - 各条件が客観的に確認できるか
   - 成功基準が明確か

## 実装橋渡しチェック

次も必ず確認してください。

- 実装者がこの `spec.md` だけで着手できるか
- `範囲内` と `範囲外` が衝突していないか
- `制約条件` が既存互換や禁止事項を十分に表しているか
- `受け入れ条件` が具体的で、テストに落とせるか
- `最小確認対象` が具体的で、最初に見るファイル・テスト・コマンドを十分狭く示せているか
- `リスクと未決定事項` に残すべき未解決が、仕様本文へ紛れ込んでいないか
- 一般ルールと具体例、期待値、補足例が互いに矛盾していないか
- ある `受け入れ条件` が、他の `受け入れ条件` や `制約条件` で必要な変更を打ち消していないか
- sanitize / slug / 正規化 / fallback を含む仕様で、全無効入力や空入力が空の識別子・不正な path・危険な key を生まないか
- 「現状の実装」「既存挙動」「現在の helper」などの記述がある場合、repo のコードやテストと矛盾していないか
- 狭いタスクなら、関連ファイル・関連テストの最小集合が見える内容になっているか。不要に repo 全体探索を前提にしていないか
- helper や current behavior の説明に、helper 自体がしていない前処理・後処理が混ざっていないか
- 狭い docs / text-only タスクなら、各変更箇所の変更後文言が literal に特定できるか。共通パターンや要約を書く場合も、それが補助説明なのか、実際の変更後文言なのかが明確か

## 出力JSON形式

以下の形式でJSONを出力してください：

```json
{
  "version": 1,
  "run_id": "{RUN_ID}",
  "created_at": "UTCタイムスタンプ",
  "updated_at": "UTCタイムスタンプ",
  "started_at": "UTCタイムスタンプ",
  "first_artifact_at": "UTCタイムスタンプ",
  "completed_at": "UTCタイムスタンプ",
  "project_root": "{PROJECT_ROOT}",
  "source": {
    "type": "spec",
    "value": "仕様書タイトル"
  },
  "summary": {
    "overall_quality": "high|medium|low",
    "headline": "全体評価の1行要約",
    "finding_count": {
      "high": 0,
      "medium": 0,
      "low": 0
    }
  },
  "findings": [
    {
      "id": "F1",
      "severity": "high|medium|low",
      "section": "完全性|明確性|一貫性|実現可能性|テスト可能性",
      "title": "指摘タイトル",
      "detail": "詳細な説明",
      "suggestion": "改善提案"
    }
  ],
  "checked_paths": ["path/to/spec.md", "path/to/verified/file.txt"],
  "checked_lines": [{"path": "path/to/file.md", "lines": "1-42"}]
}
```

**重要**: `checked_paths` と `checked_lines` は必須項目です。
- `checked_paths`: レビューで確認したファイルパスの一覧（文字列配列）。少なくとも spec.md 自体を含めてください。0 findings のときも空配列にしないでください。
- `checked_lines`: レビューで確認したファイルと行範囲の一覧（オブジェクト配列）。各オブジェクトは `{ "path": "path/to/file", "lines": "1-42" }` 形式。`lines` は人が読める範囲表記（例: `"1-42"`, `"15,30-35"`）。0 findings のときも空配列にしないでください。

## 重要度の目安

- **high**: 仕様書の核心に関わる欠陥、実装に重大な影響を与える問題
- **medium**: 改善が望ましい問題、実装に影響する可能性がある問題
- **low**: 軽微な問題、記述の改善提案

特に次は原則 `high` または `medium` にしてください。

- `spec.md` を読んでも実装者が追加質問なしで着手できない曖昧さ
- `受け入れ条件` を満たしたか判定できない記述
- ある `受け入れ条件` が、他の `受け入れ条件` で必要な変更を禁止しているケース
- `範囲内` / `範囲外` / `制約条件` の矛盾
- 失敗系や既存互換に触れるべきなのに、完全に抜けているケース
- 本文ルールと例・期待値の矛盾
- sanitize / slug / fallback の結果が空識別子や不正な path を作り得るのに、guard や扱いが未定義なケース
- repo で確認していない current behavior を本文で断定しており、実装者を誤誘導し得るケース
- 狭いタスクなのに関連ファイルや根拠が絞れておらず、実装者に不要な repo 全体探索を強いるケース
- `最小確認対象` がなく、または抽象的すぎて、実装者や reviewer が最初の working set を特定できないケース
- helper がしていない前処理（例: 翻訳）や後処理を helper/current behavior の説明に混ぜ、実装者を誤誘導し得るケース
- 狭い docs / text-only タスクなのに、変更後の literal な文言が箇所ごとに特定できず、実装者が要約や共通パターンから推測するしかないケース
- 共通の置換パターンや要約説明が、箇所別の literal な変更後文言より優先されるように読め、実装者を誤誘導し得るケース

## ステータス管理

進捗に応じて以下のJSONを更新してください:
保存先: `{STATUS_FILE}`

### レビュー開始時

```json
{
  "status": "in_progress",
  "phase": "loading",
  "progress": "仕様書のレビューを開始",
  "started_at": "UTCタイムスタンプ",
  "updated_at": "UTCタイムスタンプ"
}
```

**重要**: `phase` は進行状況に応じて更新してください:
- `loading`: 仕様書と関連ファイルの読み込み中
- `checking`: レビュー実施中
- `writing`: レビュー結果の書き込み中
- `completed`: レビュー完了

### 各セクション確認時

progress を適宜更新（例: "完全性チェック完了"、"明確性チェック完了"）

### 完了時

```json
{
  "status": "completed",
  "phase": "completed",
  "progress": "仕様書レビュー完了",
  "started_at": "レビュー開始時の時刻",
  "first_artifact_at": "最初の有効artifact作成時の時刻",
  "completed_at": "UTCタイムスタンプ"
}
```

### エラー時

```json
{
  "status": "failed",
  "progress": "エラー内容",
  "updated_at": "UTCタイムスタンプ"
}
```

## 下書き作成方法（中断防止）

レビュー結果を `{REVIEW_PATH}` に段階的に書き込んでください。すべてを一度に出そうとしないでください。

手順:
1. 開始時に `{STATUS_FILE}` を `in_progress` にし、`{REVIEW_PATH}` に findings 空配列の初期JSONを保存してください
2. 各観点の確認後に、必要な findings を `{REVIEW_PATH}` へ追記してください
3. 最後に `summary` を確定し、完成版 JSON で `{REVIEW_PATH}` を上書きしてください
4. 完了時に `{STATUS_FILE}` を completed に更新してください

初期JSONは最終形式と同じキーを使い、`findings` は空配列、`summary.finding_count` は 0 埋めから開始して構いません。

## 完了時の手順

1. {REVIEW_PATH} にレビュー結果JSONを保存してください
2. {STATUS_FILE} で指定されたファイルを completed に更新してください
3. 「レビュー完了」とのみ報告してください
